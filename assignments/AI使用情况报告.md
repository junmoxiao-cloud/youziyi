# AI 使用情况报告

## 1. 使用概况

"游子衣"项目不是单纯把 AI 当作代码生成器，而是把它放进了完整的研发流程中，承担了规划、拆解、文档整理、技术对比、答辩准备和商业分析等多种角色。

本项目中，AI 的主要使用场景包括：
- 项目初期的愿景澄清与架构规划。
- 核心功能的任务拆解与技术路线分析。
- 规则体系和文档体系的持续整理。
- 商业计划书与答辩材料的结构化生成。
- 多 Agent 协作方法的总结与复盘。

从结果上看，AI 参与的不只是"写代码"，更重要的是帮助我把一个跨 Web、小程序、后端和答辩材料的综合性项目，组织成一套可以持续推进的工作流。

## 2. Prompt 工程

### 2.1 Rules 作为系统级约束

我在 `.trae/rules` 下编写了宏观指导、技术开发规则和 API 接口规范等规则文件，把项目的核心目标、技术边界和交付要求转化为 AI 的长期上下文。这样做的价值在于：
- 保证 AI 始终围绕"亲情记录与智能关怀"展开，而不是偏离成普通社交产品。
- 保证 Web、大屏、小程序、后端接口、部署文档等输出在同一套叙事下推进。
- 降低长对话中的上下文漂移风险。

### 2.2 系统提示词与用户提示词的权重区分

在本项目中，我逐渐意识到，通用模型本身并不缺"知识"，真正影响输出质量的是信息的组织方式。因此我把提示词分成两层：
- **系统提示词 / Rules**：负责定义角色、边界、项目愿景和输出规范。
- **用户提示词**：负责提出当下的具体任务，例如"生成部署文档""补建商业计划书""重写答辩材料"。

这种做法的核心不是反复补充知识，而是进行信息流编排。也就是先让 AI 明白"站在什么位置思考"，再让它处理"眼前要做什么"。

## 3. 问题拆解

### 3.1 复杂功能拆成可执行子任务

项目中的几个核心模块都不是一句话能直接写完的，我借助 AI 做了多轮拆解：
- **超时无互动预警**：拆成"最后交互时间记录 -> Redis 缓存 -> 阈值判断 -> 查询接口 -> 后续主动通知演进"。
- **城市天气瓶**：拆成"天气数据来源 -> 视觉映射 -> Web 端动效实现 -> 降级策略 -> 与业务数据解耦"。
- **语音接龙**：拆成"录音 -> 上传 -> 存储 -> 列表查询 -> 顺序播放 -> 后续导出"。
- **答辩材料**：拆成"PPT 文字稿 -> 演示脚本 -> 问答准备稿 -> AI 使用情况报告 -> 商业计划书 -> 提交清单"。

### 3.2 以任务为导向的迭代方式

相比一次性生成整套内容，我更常用的方法是"做完一个任务，立刻总结一次经验，再进入下一轮"。这种节奏有两个好处：
- 每一轮都能快速校正方向，避免大段返工。
- 经验会沉淀到下一轮提示词和规则文件里，形成稳定的工作方法。

## 4. 多 Agent 协调

### 4.1 专用子 Agent 的作用

当任务超出纯技术实现范畴时，我没有继续用一个通用 Agent 硬顶，而是引入了专用的"智能关怀商业顾问"子 Agent。它主要承担两类工作：
- 从商业视角分析"银发经济"赛道、目标用户、MVP 公测和合规风险。
- 把模糊的"商业落地"问题拆成更容易执行的子问题，例如用户画像、推广渠道、弱实名策略、收费路径。

### 4.2 五种多 Agent 协调模式

结合实际使用和资料阅读，我总结了五种多 Agent 协调模式：
1. **生成 - 验证**：一个 Agent 负责产出，另一个 Agent 负责检查缺陷或补强。
2. **编排 - 子 Agent**：主 Agent 统筹目标，子 Agent 分别处理商业、代码、文档等垂直问题。
3. **Agent 团队**：多个 Agent 分工协作，适合大任务并行推进。
4. **消息总线**：适合多个节点围绕统一任务交换中间结果。
5. **共享状态**：适合多轮任务在统一上下文下逐步推进。

在"游子衣"里，最常用的是"编排 - 子 Agent"和"生成 - 验证"两种模式：前者用于跨领域任务拆分，后者用于文档和代码质量复核。

## 5. 人机协作反思

### 5.1 AI 擅长的是铺开思路，人类负责收口

AI 的优势在于：
- 快速整理已有信息。
- 生成多个可比较的方案。
- 在跨技术、文档、商业的混合任务中保持较高产出速度。

但最终真正决定质量的，仍然是人的审查和收口。比如：
- 是否真的符合"游子衣"的亲情叙事。
- 文档口径是否与代码现状一致。
- 技术方案是否经得起答辩追问。
- 哪些内容能保留，哪些只是 AI 的"看起来合理"。

### 5.2 Rules 与子 Agent 的边界

这次项目让我更清楚地认识到：
- **Rules 是底座**：它决定项目不跑偏，负责长期一致性。
- **子 Agent 是利刃**：它负责在特定方向上做深度突破，尤其适合商业分析、复杂拆解和专项研究。

两者不是替代关系，而是稳定框架和专项突破之间的互补关系。

## 6. 效能提升

AI 协作带来的提效主要体现在以下几个方面：
- **规划效率提升**：原本需要多轮手写整理的项目结构、交付物和任务分组，可以更快成型。
- **文档产出效率提升**：部署指南、答辩稿、问答准备稿、AI 报告和商业计划书都能在统一口径下快速生成初稿。
- **问题定位效率提升**：面对"公网为什么仍不可访问""预警系统为什么要用 Redis"这类问题，可以更快得到结构化解释。
- **跨领域补位**：技术开发者原本不擅长的商业规划与答辩叙事，也能借助 AI 快速补齐。

具体到答辩展示部分，AI 的价值不是替代思考，而是帮助我把"技术实现""产品叙事""商业逻辑""AI 协作方法"整合成一个更完整的期末作业交付包。

## 7. 结论

本项目最大的收获，不是"用了多少次 AI"，而是逐步形成了一套更稳定的协作方法：
- 先用 Rules 锁定愿景、规范和边界。
- 再用结构化 Prompt 驱动单次任务。
- 遇到垂直领域问题时，使用专用子 Agent 做深度分析。
- 每完成一轮任务，就做一次经验总结，沉淀到下一轮信息流中。

因此，AI 在"游子衣"项目中的角色可以概括为一句话：

**它不是代替开发者完成项目，而是帮助开发者更快地组织信息、拆解问题、生成初稿，并把更多精力留给最终判断与质量把控。**
