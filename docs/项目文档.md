## 1. 项目概述

### 1.1 产品定位

游子衣（YouZiYi）——「亲情的记录和见证之所」，是一个面向异地家庭亲情陪伴的产品，核心场景是子女与年迈父母之间的日常关怀连接。产品名取自古诗《游子吟》中「慈母手中线，游子身上衣」的意境，强调跨越距离的温度连接，承担「贴心管家」的角色，把老人每日的心情、健康指标、天气与语音，通过细腻的视觉与多模态反馈，沉淀成家庭成员共同的见证。

### 1.2 目标用户

- **远在他乡的子女（child）**：长期在外工作、无法时时陪伴在父母身边，希望用低门槛方式感知父母今日状态、回复问候、收到预警。
- **留守或独居老人（elder）**：身体机能下降、对智能设备有畏惧感，希望用极简操作完成每日「报平安」，并对子女的关心给出反馈。

项目采用真实多账户体系，用户角色仅允许 `elder` 与 `child` 两种枚举（见 `packages/types/src/index.ts` 的 `UserRole` 与 `apps/server/src/index.ts` 中注册校验逻辑）。禁止在前端写死测试账户或伪造家庭成员身份。

### 1.3 三端协同价值

游子衣采用「三端协同 + 后端支撑」的 Monorepo 架构，各端承担不同职责：

- **Web 端（`apps/web`）**：以 Vite + React 为母版，承担答辩大屏展示、3D 天气瓶动效、家庭全景陪伴大屏、健康趋势图表与语音接龙时间线。生产环境部署在 `https://www.youziyi.com`，继续使用相对路径 `/api`，由网关反代到后端。
- **微信小程序端（`apps/miniapp`）**：原生微信小程序开发，承担 MVP 适老落地，针对老年用户做深度适老化定制（大字号、高对比度、48×48 dp 热区）。生产环境直接请求 `https://api.youziyi.com`。
- **鸿蒙端（`apps/harmony`）**：基于 HarmonyOS NEXT 的 ArkTS + ArkUI 声明式范式，承担原生体验与系统能力调用（网络、麦克风、文件、提示）。生产环境直接请求 `https://api.youziyi.com`。鸿蒙端是本次提交的重点端，详见第 4、5、6 章。
- **后端服务（`apps/server`）**：Node.js + Express + Prisma ORM，对外统一暴露在 `https://api.youziyi.com`，本地联调默认监听 `3001` 端口。

### 1.4 公网域名约定

依据 `README.md` 与 `.trae/rules/API接口规范.md` 第 1.1.1 节，公网域名约定如下：

| 域名 | 用途 |
| --- | --- |
| `https://www.youziyi.com` | Web 前端正式访问域名；浏览器访问该域名打开 Web 前端页面。Web 生产继续使用相对路径 `/api`，由 `www.youziyi.com` 所在网关或 Nginx 反代到后端服务。 |
| `https://api.youziyi.com` | 统一公网后端 API 域名，供 Web 反代请求、微信小程序与鸿蒙原生端共同访问。 |

微信小程序发布前必须在微信公众平台将 `https://api.youziyi.com` 配置为合法 `request` 域名；鸿蒙端则在 `apps/harmony/entry/src/main/ets/common/AppConfig.ets` 中以 `CLOUD_API_BASE_URL = 'https://api.youziyi.com'` 作为生产基址。若任一端入口或反代策略变化，必须同步更新 README、Rules、API 文档与对应端配置，禁止出现分叉。

## 2. 需求分析

### 2.1 老年人陪伴痛点

1. **数字鸿沟**：老年用户对复杂表单、嵌套菜单、过小字号高度畏惧，常因「看不清、点不准、不知道下一步」而放弃使用。
2. **操作门槛**：现有家庭关怀类 App 多沿用年轻态交互范式，按钮热区过小、点击反馈单一，老人误触率高。
3. **情感疏离**：异地子女与父母之间缺乏日常低门槛的「报平安」锚点，长期缺失会形成情感断裂，突发情况下也无人察觉。

游子衣以「极低门槛 + 多模态反馈」为核心命题回应上述痛点，强制要求核心字号 24 px+、最小字号 18 px+、热区 ≥ 48×48 dp、点击关键操作必须有「视觉动效 + 物理震动（如适用） + 语音播报」的多重反馈。

### 2.2 家庭连接需求

- **真实多账户体系**：每个用户注册后获得独立 `userId`，禁止前端写死 `user_001` 之类的测试账户。登录后服务端返回 `userId` 与 `role`，作为身份判断唯一依据。
- **牵挂码配对**：长辈或子女任一方先创建家庭，服务端生成 6 位随机牵挂码（`connectionCode`），另一方输入牵挂码加入家庭，建立 `FamilyMember` 关联。
- **连接成功反馈**：加入家庭操作完成后，UI 必须展示明确的「连接成功」反馈，并展示已连接的家人信息，确认闭环后再进入主面板。
- **状态回写**：家庭创建或加入成功后，前端必须立即重新请求 `GET /api/user/profile/:userId`，以服务端最新返回的 `familyId` 与 `familyInfo` 作为「连接成功后稳定展示」的唯一依据，禁止用提交前缓存状态假定已连接完成。

### 2.3 预警需求

后端需记录双方最后一次有效互动的时间戳（`User.lastInteractionTime`），并按以下阈值触发分级告警：

| warningLevel | 触发条件 | 含义 |
| --- | --- | --- |
| 0 | < 24 小时 | 正常 |
| 1 | ≥ 24 小时 | 24 小时未互动，触发微信服务通知 |
| 2 | ≥ 36 小时 | 36 小时未互动，触发强弹窗推送 |
| 3 | ≥ 48 小时 | 48 小时未互动，自动触发短信/语音电话 |

预警状态机由后端驱动（Redis 过期回调或 Cron Job），前端只负责消费 `GET /api/warning/status/:userId` 返回的 `warningLevel` 与 `isTriggered` 字段并展示。

### 2.4 适老化需求

依据 `.trae/rules/技术开发规则.md` 第 3.1 节，所有 UI 组件须遵循：

- **字号**：最小字号不低于 18 px，核心信息（心情、提醒）字号 24 px+。
- **对比度**：前景色与背景色对比度符合 WCAG AA 级别（至少 4.5:1）。
- **点击热区**：可交互元素热区不小于 48×48 dp，元素间留有安全边距。
- **多模态反馈**：打卡、录音、收到消息等关键操作必须提供「视觉动效 + 物理震动（如适用） + 语音播报」的多重反馈。

### 2.5 每日健康采集需求

依据 `.trae/rules/数据采集与路由规范.md`，每日健康采集采用「Onboarding 配置 → 动态生成表单」的两段式机制：

- **首次用户信息采集（Onboarding）**：注册后首次登录或 `trackedMetrics` 为空时强制触发，采集 `cityCode` 与 `trackedMetrics`，写入 `User` 表作为长期配置。
- **每日信息采集（Daily Check-in）**：进入长辈关怀模式后，根据 `trackedMetrics` 动态生成表单。
  - `mood`（心情，必选，枚举 `happy` / `calm` / `sad`）
  - `steps`（步数，必选/默认）
  - `heartRate`（心率）
  - `bloodPressure`（血压）
  - `bloodSugar`（血糖）
  - `sleep`（睡眠）
- **数据隔离**：账号持久化属性（Profile）与每日动态数据（Daily Record）在数据库与前端状态管理中严格分离。`User.trackedMetrics` 为配置，`CheckIn` / `HealthRecord` 为时序数据。

为避免「高血压老人看到血糖填写框」这类无关医疗术语干扰，未启用的指标不应在表单中渲染，亦不应在上送 `metricsData` 时强行写入空值。

### 3.3 数据流向

前端各端尽量依赖服务端的「单一真实数据源（Single Source of Truth）」：

- 老人心情、步数变化实时或批量同步至服务端，写入 `CheckIn` 与 `HealthRecord` 表。
- `User.lastInteractionTime` 字段在打卡、语音上传、健康确认等关键交互时更新，并同步写入 Redis 缓存 `lastInteraction:<userId>`。
- 前端组件不读取本地缓存或静态 mock 来展示「当天健康状态」，统一以 `GET /api/health/checkin-status/:userId` 与 `GET /api/health/checkins/daily/:userId` 的服务端聚合结果为准。
- 子女端查看家庭对方当天状态时，必须先从 `GET /api/user/profile/:userId` 读取 `data.familyInfo.members[].userId`，再以「对方 userId」请求打卡状态接口，禁止用当前登录人的 `userId` 代替家庭共享对象。

### 3.4 路由守卫逻辑

依据 `apps/web/src/routing.ts` 与 `apps/harmony/entry/src/main/ets/common/AppSession.ets`，登录后首跳由 `trackedMetrics` 与 `familyId` 两个守卫字段决定，且必须收敛到共享判断函数中，禁止在多个页面各自维护冲突跳转分支。

Web 端 `resolveAuthenticatedLandingPath` 实现如下：

```ts
export function resolveAuthenticatedLandingPath(
  profile: RouteGuardProfile | null | undefined
): '/onboarding' | '/family/join' | '/role-select' {
  if (!hasCompletedOnboarding(profile)) {
    return '/onboarding';
  }
  if (!hasJoinedFamily(profile)) {
    return '/family/join';
  }
  return '/role-select';
}
```

鸿蒙端 `resolveAuthenticatedRoute` 实现等价逻辑，返回值为页面路径：

```ts
export function resolveAuthenticatedRoute(profile: UserProfileResponse | null): string {
  if (!isOnboardingCompleted(profile)) {
    return 'pages/Onboarding'
  }
  if (!hasJoinedFamily(profile)) {
    return 'pages/FamilyJoin'
  }
  return 'pages/Main'
}
```

完整流转如下：

```text
/welcome (登录/注册)
   │ 登录成功，检查 userProfile.trackedMetrics
   ▼
┌─────────────────────────┐
│ trackedMetrics 不存在/空 │ ──► 强制跳转 /onboarding (首次信息采集页)
└─────────────────────────┘
   │ trackedMetrics 已存在
   ▼
┌─────────────────────────┐
│ 检查 familyId 是否存在   │ ──► 跳转 /family/join (家庭连接页)
└─────────────────────────┘
   │ familyId 已存在
   ▼
跳转至主面板 (/companion 或 /care 或 /role-select)
```

守卫字段真实性约束：`trackedMetrics` 与 `familyId` 只有在服务端成功返回后才能判定为「已完成」。接口失败、超时或异常时，前端必须保持保守空态，禁止通过 mock 默认值或兜底数据伪造「已完成 Onboarding」或「已加入家庭」状态。

### 3.5 共享真源机制

- **城市码与标签映射**：单一真源固定为 `packages/types/src/index.ts` 中的 `CITY_OPTIONS` 与 `CITY_LABELS`。`apps/web` 与 `apps/server` 直接消费共享定义；`apps/harmony` 只允许消费单向同步产物 `apps/harmony/entry/src/main/ets/common/synced/SharedCityCatalog.ets`，禁止在 `ApiModels.ets`、页面或 mock 中再次维护 `BEIJING`、`SHANGHAI` 等分叉常量。
- **共享常量清单**：当前真源涵盖 6 个城市：`SHANGHAI / 上海`、`BEIJING / 北京`、`GUANGZHOU / 广州`、`SHENZHEN / 深圳`、`HANGZHOU / 杭州`、`CHENGDU / 成都`。
- **鸿蒙同步产物**：`SharedCityCatalog.ets` 文件首部注释明确标注 `Synced from packages/types/src/index.ts.`，导出 `SHARED_CITY_OPTIONS`、`SHARED_CITY_LABELS` 与 `resolveSharedCityLabel` 函数，与 TS 侧 `resolveCityLabel` 行为一致。
- **业务时区与打卡窗口**：`packages/types/src/index.ts` 还导出 `BUSINESS_TIMEZONE = 'Asia/Shanghai'`、`DAILY_CHECKIN_WINDOW_RULES`（早间 5–12 时、晚间 17–23 时、整体窗口 5–23 时）、`resolveCheckInPeriod`、`resolveTodayCheckInWindowPolicy`、`buildDailyHealthAggregates` 等函数，Web 与 Server 共用，鸿蒙通过接口返回的 `window` 字段复用同一策略，不单独维护时间边界。

## 4. 功能模块说明

本章节逐模块说明功能、涉及接口与涉及端。所有接口路径与字段均取自 `.trae/rules/API接口规范.md` 与 `apps/server/src/index.ts` 实际实现。

### 4.1 登录注册

- **功能**：建立真实多账户体系，注册时必填 `name`、`password`、`role`，`role` 仅允许 `elder` 或 `child`；登录成功后返回 `userId` 与 `role`，前端据此建立会话。
- **涉及接口**：
  - `POST /api/auth/register`，请求体 `{ name, password, role, cityCode? }`，返回 `{ userId, name, role }`。
  - `POST /api/auth/login`，请求体 `{ name, password }`，返回 `{ userId, name, cityCode, role }`。
- **涉及端**：Web、小程序、鸿蒙。
- **鸿蒙实现**：`apps/harmony/entry/src/main/ets/services/AppRepository.ets` 的 `register` / `login` 方法封装上述接口，登录成功后通过 `appSession.setAuth(userId, name, role)` 写入会话，再调用 `getUserProfile` 拉取资料并交给 `resolveAuthenticatedRoute` 决定首跳。
- **服务端校验**：`apps/server/src/index.ts` 在 `/api/auth/register` 中显式校验 `role !== 'elder' && role !== 'child'` 返回 400，并校验用户名唯一性。

### 4.2 Onboarding 首次信息采集

- **功能**：用户注册后首次登录或 `trackedMetrics` 为空时强制触发，采集 `cityCode` 与 `trackedMetrics`，写入 `User` 表作为长期配置。
- **涉及接口**：
  - `POST /api/user/profile/update`，请求体 `{ userId, cityCode?, city?, trackedMetrics? }`，返回更新后的用户资料。
  - `GET /api/user/profile/:userId`，路由守卫读取 `trackedMetrics` 判空。
- **涉及端**：Web、鸿蒙。
- **鸿蒙实现**：`apps/harmony/entry/src/main/ets/pages/Onboarding.ets` 引用 `SharedCityOption` 渲染城市选项，引用 `METRIC_OPTIONS` 渲染指标多选；`mood` 指标默认选中且不可取消（与 `DEFAULT_TRACKED_METRICS = ['mood']` 一致）。
- **守卫字段真实性**：`trackedMetrics` 须以服务端返回的归一化结果为准。服务端 `normalizeTrackedMetrics` 函数会自动剥离非法值，且当结果为空时拒绝更新并返回 400「当前阶段仅支持标准健康指标，请至少保留心情后再保存」。

### 4.3 家庭连接

- **功能**：长辈或子女任一方创建家庭，服务端生成 6 位随机牵挂码（`connectionCode`），另一方输入牵挂码加入；连接成功后服务端返回 `familyId` 与家庭成员列表。
- **涉及接口**：
  - `POST /api/family/create`，请求体 `{ userId, familyName? }`，返回 `{ familyId, inviteCode, name }`。若用户已在家庭中，直接返回现有牵挂码。
  - `POST /api/family/join`，请求体 `{ userId, inviteCode }`，返回 `{ familyId, name }`。若该账号已加入其他家庭返回 400。
  - `GET /api/family/:familyId`，查询家庭详情（鸿蒙第一阶段依赖接口清单中列出）。
  - `GET /api/user/profile/:userId`，连接成功后立即重新请求，以服务端最新 `familyId` 与 `familyInfo` 为准。
- **涉及端**：Web、鸿蒙。
- **鸿蒙实现**：`apps/harmony/entry/src/main/ets/pages/FamilyJoin.ets` 同时承担「创建」与「加入」两种模式，`aboutToAppear` 时主动调用 `getUserProfile` 回填已连接家庭信息；`onBackPress` 在已连接状态下拦截返回并跳回 `pages/Main`。
- **闭环要求**：连接成功后 UI 必须展示明确的「连接成功」反馈，并展示已连接的家人信息，确认闭环后再进入主面板。禁止直接用提交前缓存状态假定已连接完成。

### 4.4 每日打卡

- **功能**：长辈每日进入打卡面板，根据 Onboarding 配置的 `trackedMetrics` 动态生成表单，提交心情、步数及可选指标；服务端按业务日（`Asia/Shanghai`）聚合今日摘要、最近几天趋势与时段窗口策略。
- **涉及接口**：
  - `GET /api/health/checkin-status/:userId`：返回今日是否已打卡、当前时段、表单初始值、今日摘要、`window` 窗口策略对象。
  - `POST /api/health/checkin`：提交打卡，请求体 `{ userId, mood, steps, heartRate?, timestamp, metricsData? }`。
  - `GET /api/health/checkins/daily/:userId?days=7&date=YYYY-MM-DD`：返回按天聚合结构，包含 `today` / `yesterday` / `recentDays[]`。
- **涉及端**：Web、鸿蒙（小程序离线缓存能力规划中）。
- **Web 实现**：`apps/web/src/pages/CareDashboard.tsx` 基于 `todayCheckInStatus.form.editableMetrics` 动态渲染心情、步数、心率、血压、血糖、睡眠卡片，并基于 `window.promptState` 展示对应提示文案；提交成功后立即刷新 `checkin-status` 与 `daily` 两个接口。
- **鸿蒙实现**：`apps/harmony/entry/src/main/ets/pages/tabs/HomeTab.ets` 是打卡入口，引用 `HOME_CHECKIN_PAGE_DEFINITION` 与 `HomeCheckInState`，按 `todayStatus` 渲染表单与窗口提示；`appRepository.submitCheckIn` 调用 `POST /api/health/checkin`。
- **字段说明**：
  - `window.currentPeriod` 取值 `morning | daytime | evening | closed`。
  - `window.promptState` 取值 `before_window | morning_checkin | daytime_checkin | evening_checkin | already_checked_in | missed_window`。
  - `form.editableMetrics` 优先级高于前端本地默认值；`form.initialValues` 通常来自最近一次记录，可做最小预填。
- **千人千面**：高血压老人只会看到血压填写框，糖尿病老人只会看到血糖填写框，无关指标不上送空值。

### 4.5 健康预警

- **功能**：后端记录双方最后一次有效互动时间戳，按 24/36/48 小时阈值触发分级告警；前端读取预警状态并展示亲情温度计、倒计时与提醒文案。
- **涉及接口**：
  - `GET /api/warning/status/:userId`，返回 `{ lastInteractionTime, warningLevel, isTriggered }`。
- **涉及端**：Web、鸿蒙。
- **Web 实现**：`apps/web/src/pages/CompanionDashboard.tsx` 顶部根据 `warningStatus.warningLevel` 渲染「管家温馨提示（level 1）」或「管家紧急提醒（level ≥ 2）」Toast，并以「亲情温度计」圆形卡片配合脉冲动画展示「温度正好 / 需要添柴啦 / 快去暖暖场」。
- **鸿蒙实现**：`apps/harmony/entry/src/main/ets/pages/tabs/HealthTab.ets` 在 `aboutToAppear` 中并行调用 `getWarningStatus` 与 `getTodayCheckInStatus`，按 `warningLevel` 切换背景色与文案（`#E7F5EC` 温度正好 / `#FFF8E1` 需要添柴 / `#FFEBEE` 快去暖暖场）。
- **服务端实现**：`apps/server/src/index.ts` 的 `/api/warning/status/:userId` 优先从 Redis 读取 `lastInteraction:<userId>`，缺失时降级从 `User.lastInteractionTime` 读取并回写 Redis，按小时差计算 `warningLevel`。

### 4.6 天气

- **功能**：城市天气瓶动效，获取双方所在城市的天气信息；Web 端用 3D / 2D 天气瓶呈现，鸿蒙端用 `WeatherCard` 组件呈现。
- **涉及接口**：
  - `GET /api/weather/current?cityCode=<code>`，返回 `{ cityCode, cityName, weatherType, temperature, humidity }`。`weatherType` 取值 `sunny | rainy | cloudy | snowy`。
- **涉及端**：Web、鸿蒙。
- **服务端实现**：`apps/server/src/index.ts` 的 `/api/weather/current` 调用 `wttr.in` 真实接口，通过 `resolveCityLabel` 解析城市名；解析失败时降级返回默认天气数据（`weatherType: 'sunny'`、`temperature: 20`、`humidity: 50`），保证页面不空白。
- **Web 实现**：`apps/web/src/components/WeatherBottle.tsx` 按 `weatherType` 切换渐变背景、图标与文案（晴朗 / 多云 / 下雨 / 下雪）。陪伴大屏严格按 `familyInfo.members[].cityCode` 分别请求长辈与子女城市天气，禁止写死 `BEIJING` / `SHANGHAI`。当资料城市与天气结果暂时不一致时，保守显示为「天气暂不可用」。
- **鸿蒙实现**：`apps/harmony/entry/src/main/ets/pages/tabs/HomeTab.ets` 通过 `resolveFamilyWeatherList` 计算家庭成员天气列表，渲染 `WeatherCard` 组件。

### 4.7 语音接龙

- **功能**：家庭故事语音接龙，子女与长辈轮流录制语音片段并上传，组成时间线；任一方可在时间线中播放历史片段。
- **涉及接口**：
  - `POST /api/voice/upload`（`multipart/form-data`，字段 `file`、`userId`、`storyId`），返回 `{ voiceId, url, duration }`。
  - `GET /api/voice/list/:userId`，返回 `{ records: [{ id, role, timeLabel, duration, url }] }`，即使为空也必须返回 `[]`。
- **涉及端**：Web、鸿蒙。
- **Web 实现**：`apps/web/src/components/VoiceTimeline.tsx` 使用浏览器 `MediaRecorder` 录制 `audio/webm`，上传至 `/api/voice/upload`；接口失败时回退到本地 `Blob URL` 用于预览（仅作演示兜底）。卸载时调用 `mediaRecorder.stop()` 与 `audio.pause()` 清理资源。
- **鸿蒙实现**：`apps/harmony/entry/src/main/ets/pages/tabs/InteractionTab.ets` 使用 `@ohos.multimedia.media` 的 `AVRecorder` 录制（编码 `AUDIO_AAC`，码率 48000，采样率 44100，单声道，容器 `CFT_MPEG_4A`），文件写入 `getContext().cacheDir + '/voice_<timestamp>.m4a'`；播放使用 `AVPlayer`，监听 `stateChange` 事件在 `completed` / `error` 状态自动释放资源。详见第 6 章。
- **服务端实现**：`apps/server/src/index.ts` 使用 `multer` 处理上传，文件落盘到 `apps/server/public/uploads/`，并通过 `express.static` 暴露访问 URL；上传成功后更新 `User.lastInteractionTime` 与 Redis 缓存。

### 5.2 后端

- **框架**：Node.js + Express，入口 `apps/server/src/index.ts`，监听 `process.env.PORT || 3001`。
- **ORM**：Prisma Client，schema 位于 `apps/server/prisma/schema.prisma`，生成产物输出到 `apps/server/src/generated/prisma`。
- **数据库**：

## 6. 关键实现细节

### 6.1 路由守卫实现

- **Web 端**：`apps/web/src/routing.ts` 提供 `hasCompletedOnboarding`、`hasJoinedFamily`、`resolveAuthenticatedLandingPath` 三个集中式函数。`hasCompletedOnboarding` 调用 `normalizeTrackedMetrics(profile?.trackedMetrics).length > 0` 判断，自动剥离非法值与空字符串；`hasJoinedFamily` 严格校验 `familyId` 是非空字符串。所有页面共用这两个函数，禁止各自维护冲突跳转分支。
- **鸿蒙端**：`apps/harmony/entry/src/main/ets/common/AppSession.ets` 的 `isOnboardingCompleted` 同时校验城市（`cityCode` 或 `city` 至少一项非空）与 `trackedMetrics` 数组非空；`hasJoinedFamily` 直接判 `Boolean(profile?.familyId)`。`resolveAuthenticatedRoute` 返回鸿蒙页面路径 `pages/Onboarding` / `pages/FamilyJoin` / `pages/Main`。
- **守卫字段真实性**：`trackedMetrics`、`familyId` 只有在服务端成功返回后才能判定为「已完成」。接口失败、超时或异常时，前端必须保持保守空态，禁止通过 mock 默认值或兜底数据伪造「已完成 Onboarding」或「已加入家庭」状态。`Login.ets` 在 `finishAuth` 中先 `setAuth` 再 `getUserProfile`，拿到 `response.data` 后才 `setProfile` 并 `replaceUrl` 跳转，失败时弹 `promptAction.showDialog` 提示错误并保持原页面。

### 6.2 共享城市码真源

- **真源**：`packages/types/src/index.ts` 的 `CITY_OPTIONS` 数组与 `CITY_LABELS` 映射。
- **同步产物**：`apps/harmony/entry/src/main/ets/common/synced/SharedCityCatalog.ets` 文件首部注释：

  ```text
  // Synced from packages/types/src/index.ts.
  // Harmony must consume this one-way synced artifact instead of redefining city
  // constants inside page or model files.
  ```

- **单向同步**：鸿蒙只允许消费该产物，禁止在 `ApiModels.ets`、页面或 mock 中再次内联维护 `BEIJING`、`SHANGHAI` 等分叉常量。
- **Onboarding 引用**：`apps/harmony/entry/src/main/ets/pages/Onboarding.ets` 引入 `SharedCityOption` 类型，城市选项完全来自同步产物。

### 6.3 鸿蒙原生 API 调用细节

#### 6.3.1 网络请求

`AppRepository.ets` 中的 `requestRemote<T>` 是鸿蒙端唯一的网络出口：

```ts
function requestRemote<T>(
  path: string,
  method: http.RequestMethod,
  payload?: Object
): Promise<ApiResponse<T>> {
  const httpRequest = http.createHttp()
  const requestUrl = resolveApiUrl(path)
  const options: http.HttpRequestOptions = {
    method: method,
    expectDataType: http.HttpDataType.STRING,
    connectTimeout: 10000,
    readTimeout: 10000,
    header: { 'Content-Type': 'application/json' } as Object
  }
  if (payload !== undefined) {
    options.extraData = JSON.stringify(payload)
  }
  return new Promise((resolve, reject) => {
    httpRequest.request(requestUrl, options, (error, data) => {
      httpRequest.destroy()
      if (error) { reject(...); return }
      if (data.responseCode < 200 || data.responseCode >= 300) { reject(...); return }
      try { resolve(parseApiResponse<T>(data.result as string)) }
      catch (parseError) { reject(...) }
    })
  })
}
```

特点：
- 单次请求完成后立即 `httpRequest.destroy()` 释放底层资源，避免句柄泄漏。
- HTTP 状态码非 2xx 时拒绝并截取前 300 字符响应体便于排错。
- JSON 解析失败时拒绝并回写原始响应片段，方便联调定位。
- 所有路径通过 `resolveApiUrl(path)` 拼接 `APP_CONFIG.apiBaseUrl`，禁止在调用方硬编码域名。

#### 6.3.2 麦克风权限

`module.json5` 中声明：

```json5
{
  "name": "ohos.permission.MICROPHONE",
  "reason": "$string:microphone_reason",
  "usedScene": {
    "abilities": ["EntryAbility"],
    "when": "inuse"
  }
}
```

- `reason` 引用 `$string:microphone_reason`（值为「用于录制家庭语音消息，实现亲情语音接龙功能」），用于在权限弹窗向用户解释用途。
- `when: "inuse"` 表示仅在应用使用期间申请，符合最小权限原则。

#### 6.3.3 录音与播放

`InteractionTab.ets` 中录音流程：

```text
1. recordFilePath = getContext().cacheDir + '/voice_' + Date.now() + '.m4a'
2. file = await fs.open(recordFilePath, fs.OpenMode.CREATE | fs.OpenMode.READ_WRITE)
3. recordFileFd = file.fd
4. avRecorder = await media.createAVRecorder()
5. avConfig = {
     audioSourceType: AUDIO_SOURCE_TYPE_MIC,
     profile: {
       audioCodec: AUDIO_AAC,
       audioBitrate: 48000,
       audioSampleRate: 44100,
       audioChannels: 1,
       fileFormat: CFT_MPEG_4A
     },
     url: 'fd://' + recordFileFd
   }
6. await avRecorder.prepare(avConfig)
7. await avRecorder.start()
8. 启动脉冲动画 pulseTimer
```

停止流程：

```text
1. await avRecorder.stop() / release() ; avRecorder = undefined
2. fs.close(recordFileFd) ; recordFileFd = -1
3. 调用 appRepository.uploadVoice(userId, recordFilePath)
4. 上传成功后 loadVoiceList() 刷新列表
```

播放流程使用 `AVPlayer`，监听 `stateChange` 事件：状态为 `completed` 时清空 `playingId` 并 `releasePlayer`；状态为 `error` 时同样释放并 `showToast('播放出错')`。组件 `aboutToDisappear` 时统一调用 `releaseRecorder` 与 `releasePlayer`，避免后台残留。

### 6.5 预警状态机

- **后端驱动**：`apps/server/src/index.ts` 中 `User.lastInteractionTime` 字段在打卡、语音上传、健康确认时被 `prisma.user.update` 更新，并同步写入 Redis `lastInteraction:<userId>`。
- **异步计算**：`/api/warning/status/:userId` 优先读 Redis，缺失时降级读 `User.lastInteractionTime` 并回写 Redis；按小时差计算 `warningLevel`：

  ```ts
  if (hoursSinceLastInteraction >= 48) warningLevel = 3;
  else if (hoursSinceLastInteraction >= 36) warningLevel = 2;
  else if (hoursSinceLastInteraction >= 24) warningLevel = 1;
  ```

- **事件分发**（规划层，对应 `.trae/rules/技术开发规则.md` 第 4 节）：
  - Level 1（24h）：微信服务通知
  - Level 2（36h）：强弹窗推送
  - Level 3（48h）：自动触发短信/语音电话
- **前端消费**：Web 与鸿蒙均只读取 `warningLevel` 与 `isTriggered` 字段展示，不自行计算时间差。

### 6.7 Web 3D 天气瓶优雅降级

依据 `.trae/rules/技术开发规则.md` 第 3.2 节，三级降级策略：

- **一级降级（数据降级）**：当 `/api/weather/current` 调用失败时，服务端 `apps/server/src/index.ts` 已实现降级，返回 `weatherType: 'sunny'`、`temperature: 20`、`humidity: 50`、`message: '降级为默认天气数据'`，保证页面不空白。
- **二级降级（视觉降级）**：检测到设备不支持 WebGL 时，前端平滑降级为高质量 CSS 2D 动画。当前 `WeatherBottle.tsx` 即为 2D 实现版本：按 `weatherType` 切换渐变背景、图标与文案，使用 `backdrop-blur` 与多重 `blur-3xl` 光晕营造质感。
- **三级降级（效果降级）**：3D 渲染针对 `cloudy` / `rainy` 等天气状态建立多层次视觉表现（背景渐变、漂浮云层、雾感粒子），帧率过低时动态剔除粒子层。当前 2D 版本天然不涉及高耗能粒子，已规避此问题。
- **组件解耦**：`WeatherBottle` 仅接收 `weatherType: string` 单一 Prop，与业务数据完全解耦。
- **城市真实性**：陪伴大屏严格按 `familyInfo.members[].cityCode` 分别请求长辈与子女天气；若资料城市与天气结果暂时不一致（`weatherMatchesCityCode` 返回 false），保守显示为「天气暂不可用」，禁止回退为固定北京/上海假数据。

### 6.8 离线缓存与弱网处理

- **小程序端**（规划能力）：在弱网或无网状态下，老人心情打卡或语音暂存本地，待网络恢复后静默上传。
- **鸿蒙端**：当前 `apiMode` 默认 `REMOTE`，未启用 Mock 兜底；`verification.allowMockFallback` 仅在开发态生效，正式环境即使远程接口不可达也禁止自动回退 Mock，避免掩盖真实问题。
- **Web 端**：`VoiceTimeline.tsx` 中 `/api/voice/upload` 失败时回退到本地 `Blob URL` 仅作演示预览，不写入服务端；`CompanionDashboard.tsx` 中 `syncDashboard` 每 60 秒轮询一次，单次失败不阻断后续轮询。
