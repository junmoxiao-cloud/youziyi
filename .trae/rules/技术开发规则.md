# 游子衣 - 技术开发 Rules

本文件是《游子衣》项目的核心技术规范，承接宏观指导与《产品计划书》，旨在为开发过程中的技术栈选择、组件设计、状态管理与工程化实践提供明确的标准与约束。

## 上位文档约束说明

本文件属于《游子衣》项目的技术落地规则，承接上位指导文档 `docs/游子衣：亲情的记录和见证之所.md` 中对项目愿景、核心功能、交付清单、双端协同架构及整体体验基调的定义。

因此，本文件内涉及技术栈选型、组件封装、状态管理、接口约定、工程结构、性能优化与异常处理等内容时，必须以 `docs/游子衣：亲情的记录和见证之所.md` 为总纲进行展开，不得偏离其功能边界与产品目标；如本文件与该文档存在冲突，统一以上位文档为准，本文件仅负责技术细化与工程化约束。

## 1. 技术栈约束 (Tech Stack Constraints)

本项目采用双端协同与后端支撑的全栈开发模式，建议采用以下现代技术栈：

*   **Web 网页端（大屏展示与动效）**
    *   **框架**：React 生态 (Next.js)。
    *   **样式**：Tailwind CSS，配合 CSS Modules 隔离复杂组件样式。
    *   **动效与可视化**：
        *   3D渲染：React Three Fiber（用于“城市天气瓶”高精度渲染）。
        *   数据图表：ECharts / D3.js（用于全景数据大屏健康趋势展示）。
    *   **状态管理**：Zustand / Redux Toolkit。
*   **微信小程序端（MVP适老落地）**
    *   **框架**：原生微信小程序开发。
    *   **UI 库**：必须进行**深度适老化定制**（大字号、高对比度）。
*   **后端服务**
    *   **框架**：Node.js (NestJS / Express) 或 Go / Python。
    *   **数据库**：PostgreSQL / MySQL (业务数据) + Redis (预警倒计时、高频状态缓存)。
    *   **对象存储**：阿里云 OSS 或腾讯云 COS（用于存储语音接龙的音频流文件）。

## 2. 目录规范与工程结构 (Directory Structure)

推荐采用 Monorepo（如使用 pnpm workspaces 或 Turborepo）进行工程化管理，实现多端代码复用：

```text
youziyi-project/
├── apps/
│   ├── web/           # Web 端工程 (答辩大屏与动效展示)
│   ├── miniapp/       # 小程序端工程 (MVP落地、适老交互)
│   └── server/        # 后端 API、定时任务与预警逻辑
├── packages/
│   ├── shared/        # 共享工具类 (utils, constants)
│   ├── types/         # TypeScript 类型定义 (接口协议)
│   └── ui/            # 可复用的基础 UI 组件
└── docs/              # 项目交付物、答辩PPT文本、商业计划书等
```

## 3. 组件开发规范 (Component Specifications)

### 3.1 适老化组件规范 (小程序端优先)
为满足老年用户“极低门槛”的使用需求，UI 组件需严格遵循以下规范：
*   **视觉要求**：
    *   **字号**：最小字号不得低于 `18px`（或同等 rpx/rem），核心信息（如心情、提醒）字号需达到 `24px+`。
    *   **对比度**：前景色与背景色对比度需符合 WCAG AA 级别（至少 4.5:1）。避免使用相近色进行信息区分。
*   **交互与热区**：
    *   **点击热区**：可交互元素（按钮、卡片）的点击热区不得小于 `48x48 dp`，且元素间需留有充足的安全边距，防止误触。
    *   **多模态反馈**：用户的每一次关键操作（打卡、录音、收到消息），必须提供“视觉动效 + 物理震动（如适用） + 语音播报”的多重反馈。

### 3.2 动效与可视化组件规范 (Web端优先)
Web 端承担答辩大屏展示与精美互动的重任，需兼顾视觉冲击力与性能：
*   **资源与性能管理**：
    *   Three.js/WebGL 组件（如 `WeatherBottle`）在卸载时必须执行内存回收清理（`dispose` 几何体、材质和纹理）。
    *   动画循环必须依赖 `requestAnimationFrame`，避免使用 `setInterval`。
*   **优雅降级 (3级降级策略)**：
    *   **一级降级（数据降级）**：当天气/业务接口请求失败时，前端需展示结构化的默认占位数据，保证页面不空白。
    *   **二级降级（视觉降级）**：检测到设备性能不足或不支持 WebGL（如获取上下文失败）时，必须平滑降级为高质量的 CSS 2D 动画或静态图片卡片（显示温度、图标等），保证应用不崩溃。
    *   **三级降级（效果降级）**：针对 3D 渲染，如 `cloudy`、`rainy` 等天气状态，需建立多层次的视觉表现（背景渐变、漂浮云层、雾感粒子等），当设备帧率过低时，可动态剔除粒子等高耗能层。
*   **组件封装**：
    *   动效组件应与业务数据解耦，通过 Props（如 `weatherType`, `interactionEvent`）驱动视觉变化。

## 4. 状态管理与预警机制 (State & Warning System)

*   **数据流向**：前端各端尽量依赖服务端的“单一真实数据源 (Single Source of Truth)”。小程序端的心情、步数变化应实时或批量同步至服务端。
*   **超时预警状态机 (Timeout Warning System)**：
    *   **后端驱动**：后端需记录双方最后一次有效互动的时间戳。
    *   **异步计算**：通过 Redis 的过期回调 (Key-space Notifications) 或后台定时任务 (Cron Job) 扫描时间差。
    *   **事件分发**：根据 24小时/36小时/48小时 的阈值，触发不同级别的告警（微信服务通知 -> 强弹窗推送 -> 自动触发短信/语音电话）。

## 5. 代码风格与质量控制 (Code Style & Linting)

*   **命名规范**：
    *   变量与函数：`camelCase`
    *   组件与类：`PascalCase`
    *   常量与枚举：`UPPER_SNAKE_CASE`
*   **代码检测**：
    *   项目必须配置 ESLint + Prettier，并通过 Husky + lint-staged 在提交前进行自动化格式校验与拦截。
*   **Git 规范**：
    *   遵循 Conventional Commits 规范（如 `feat: 新增天气瓶动效`, `fix: 修复语音拼接时间轴错误`）。
*   **注释规范**：
    *   复杂的业务逻辑（如音频编码转换、预警触发逻辑、3D渲染计算）必须附带详尽的注释。
    *   公用接口与组件需使用 TSDoc/JSDoc 标明参数、返回值及使用说明。

## 6. 接口与异常处理 (API & Error Handling)

*   **API 设计**：
    *   遵循 RESTful 规范，接口返回统一数据结构：`{ code: 0, data: {...}, message: "success" }`。
    *   `docs/API接口规范.md` 为项目 API 协议正文、字段说明与请求/响应示例的唯一参考文档；后续新增、修改或废弃接口时，必须同步遵守并更新该文档。
    *   本地联调默认以后端 `3001` 端口为统一基准，Web 代理、小程序请求基地址、脚本配置与环境变量必须保持一致；若端口调整，必须同步修改三端配置与文档。
    *   项目采用真实多账户体系，用户角色仅允许 `elder` 与 `child` 两种枚举；认证、用户资料、家庭关系、Onboarding 与预警相关接口在涉及身份判断时，必须以服务端返回的 `userId` 和 `role` 为唯一依据。
    *   语音列表、天气、预警、家庭关系等接口的响应字段属于跨端共享协议；当 API、`packages/shared` / `packages/types`、前端 `store`、组件 Props 中任一字段新增、删除、改名或嵌套层级调整时，必须作为一次**全仓联动更新**处理。
    *   全仓联动更新至少覆盖：后端接口实现、`docs/API接口规范.md`、共享类型与常量、Web/小程序/后续鸿蒙的状态映射层、直接消费字段的组件、mock 数据、兜底默认值、联调脚本与相关 Rules，禁止只改其中一层。
    *   若接口返回结构发生包装层变化（如从 `data` 直出改为 `data.records` / `data.items`），必须同步更新 `store` 与组件的读取路径，并通过 repo 范围检索确认旧字段引用已清理。
*   **容灾与弱网处理**：
    *   小程序端需支持**离线缓存**。在弱网或无网状态下，老人的心情打卡或语音可暂存本地，待网络恢复后静默上传。
*   **错误捕获**：
    *   前端配置全局 ErrorBoundary，防止单个组件报错导致页面白屏。
    *   关键交互（如预警接口调用、音频上传）需包含重试机制。
*   **本地联调故障排查**：
    *   遇到 `ECONNREFUSED` 时，必须依次检查后端服务是否启动、是否监听在约定的 `3001` 端口、环境变量与代理是否一致，以及是否存在硬编码旧端口或旧主机地址。
    *   若近期调整过后端端口，必须同步更新 Web 代理、小程序请求基地址、启动脚本和 `docs/API接口规范.md`，禁止通过临时改某一端代码绕过统一约定。
    *   若仅某一端报 `ECONNREFUSED`，优先排查该端独有的 `.env`、本地缓存配置或调试脚本，确认其没有继续连接废弃端口。

## 7. 前端构建与性能优化 (Build & Performance)

*   **产物体积告警处理 (Chunk Size Warning)**：
    *   本项目 Web 端因包含大屏展示，需引入复杂的第三方库（如 Three.js, ECharts 等），使用 Vite 构建时极易触发默认的 `500KB` 产物告警（`Some chunks are larger than 500 kB`）。
    *   **处理规范**：
        1. **合理分包 (Chunk Splitting)**：在 `vite.config.ts` 的 `build.rollupOptions` 中配置 `manualChunks`，将 `react` 核心库、`three` 渲染库、`echarts` 等大体积依赖单独打包，避免业务代码文件过大。
        2. **调整阈值**：在确保已进行合理分包的前提下，将 `build.chunkSizeWarningLimit` 适当调高（例如调至 `1000` 或 `1500`），避免无效的控制台警告一直触发干扰视线。
*   **接口基地址与端口管理**：
    *   本地联调阶段，Web 端开发代理与后端服务端口必须统一，默认以后端 `3001` 端口为本地基准；若临时调整端口，必须同步更新环境变量与启动脚本，而不是仅修改单端代码。
    *   Vite 代理目标、线上 API 基地址、云服务器端口与反向代理地址必须通过环境变量或部署配置注入，禁止在业务代码或页面逻辑中硬编码具体云服务器端口。
    *   新增或修改认证、家庭连接、Onboarding 等接口时，若请求字段或端口约定发生变化，必须同步更新 `docs/API接口规范.md`，保持三端与后端口径一致。
